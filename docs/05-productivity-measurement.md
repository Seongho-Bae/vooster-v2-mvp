# 05 — vspec 도입의 개발생산성 측정 계획

이 문서는 *"vspec을 쓰면 작업자(사람)의 개발생산성이 실제로 올라가는가?"*를
검증하기 위한 측정 설계다. 생산성의 주어는 에이전트가 아니라 **사람**이다 —
에이전트는 도구이고, 측정 대상은 그 도구를 모는 작업자의 생산성이다. 프레임워크는
_Software Engineering at Google_ 7장의 **QUANTS**(생산성의 5개 트레이드오프 차원,
원래 엔지니어=사람 생산성 프레임)와 **GSM**(Goal → Signal → Metric)을 이 프로젝트
맥락에 맞춰 적용한다.

> 산출물 범위: **이 계획 문서까지**. CLI 텔레메트리 계측, 설문 키트, 평가용 과제
> 세트 등 후속 준비물은 별도 논의로 결정한다. (§8 참조)

---

## 1. 핵심 가설과 측정 단위

**가설 (H1).** vspec으로 작성한 Cockburn 유스케이스를 입력으로 주면, **작업자가**
스펙 없음 / 손작성 / Superpowers 대비 **더 적은 개입·검증부담으로 더 정확하게
목표를 완수하고, 산출물을 더 신뢰한다.** 에이전트의 코딩 속도·토큰은 부차적
지표(가드레일)다 — 어차피 손은 에이전트가 움직이므로, 사람을 덜 귀찮게 하면서
정확히 끝내는 것이 핵심이다.

**측정 단위는 "스펙"이 아니라 "스펙 → 구현" 전체 흐름이다.** 스펙을 작성하는
이유는 결국 구현을 잘하기 위해서이므로, 작성 품질이나 작성 속도만 보는 측정은
무의미하다. 따라서 모든 과제는 두 페이즈로 구성한다.

```
[Phase A] 스펙 작성  ──►  [Phase B] 구현
   (N·일부 Q)              (Q·사람부담의 본체)
```

생산성은 단일 스칼라가 아니라 **2축**으로 본다.

- **품질 축 (Q):** 구현이 얼마나 정확·완전하게 목표를 달성하는가 (헤드라인 결과지표)
- **사람 부담 축:** 작업자가 흐름에 얼마나 끌려 들어가는가 — 개입·검증부담·신뢰
  (헤드라인 비용지표). 단일 숫자로 합치지 않고 3개 지표를 고정 패널로 함께 본다(§4 A/S).

> 기계 비용(end-to-end 시간·토큰)은 이 축에서 빼고 **가드레일**로만 둔다(§5–6).
> "더 빠른데 더 틀린/사람을 더 귀찮게 한" 결과를 한 숫자로 뭉개지 않는다. 항상
> (품질, 사람 부담) 평면 위에서 4개 군을 비교한다.

## 2. 측정 철학 (왜 이렇게 하는가)

책의 7장이 강조하는 세 가지를 그대로 가져온다.

1. **트레이드오프부터 의심한다.** vspec의 가장 그럴듯한 시나리오는
   _Phase A(작성)에 시간을 더 쓰는 대신 Phase B(구현)의 정확성을 높이고 사람
   개입·검증부담을 줄이는_ 것이다. 한 차원만 보면 정반대 결론이 난다. → QUANTS
   5개를 **함께** 본다.
2. **측정할 가치가 있는가를 먼저 묻는다.** 각 지표는 "이 값이 바뀌면 우리가
   행동(=vspec을 밀지/접을지/고칠지)을 바꾸는가?"를 통과해야 한다. 통과 못 하는
   지표는 계측하지 않는다.
3. **결과를 사전 등록한다(pre-registration).** "vspec이 이겼다"의 기준을 데이터
   보기 **전에** 못 박는다(§6). 사후에 이긴 지표만 고르는 것을 막는다.

## 3. 실험 설계

### 3.1 비교군 (4 arms)

같은 과제 프롬프트를 4가지 워크플로우로 처리한다. **과제 프롬프트는 모든 군에서
동일**하고, 오직 "스펙 워크플로우"만 다르다.

| ID   | 군                  | Phase A (작성)                                       | Phase B (구현)                 |
| ---- | ------------------- | ---------------------------------------------------- | ------------------------------ |
| `C0` | 스펙 없음 (control) | — (생략)                                             | 과제 프롬프트만 보고 바로 구현 |
| `C1` | 손작성              | 자유형식 마크다운 스펙을 직접 작성                   | 그 스펙을 입력으로 구현        |
| `C2` | Superpowers         | Superpowers 플러그인의 스펙/플래닝 워크플로우        | 그 산출물을 입력으로 구현      |
| `T`  | **vspec** (처치군)  | `vspec`으로 Cockburn 유스케이스 작성 + `doctor` 통과 | 그 유스케이스를 입력으로 구현  |

> 가정: `C2`의 "Superpowers"는 Claude Code의 Superpowers 플러그인(스펙/플래닝
> 스킬 워크플로우)을 가리킨다고 본다. 다른 도구를 의도했다면 알려주면 교체한다.

### 3.2 과제 세트 (예시 입력)

- 표준 코딩 과제 6~10개. 크기를 섞는다(소: 단일 함수/버그픽스, 중: 다단계 기능,
  대: 엣지케이스가 많은 기능).
- **각 과제는 "숨겨진 골든 수용 테스트"를 동반**한다. 에이전트에게는 과제
  서술만 주고, 평가는 에이전트가 못 본 수용 테스트로 채점한다.
- 과제 서술에는 **명시적 엣지케이스를 의도적으로 일부만** 적는다. vspec의
  Extensions/guarantees 구조가 "서술에 없던 엣지케이스를 구현이 잡아내는가"에
  기여하는지 보기 위함이다.
- 이 프로젝트 도메인(예: 파서/검증/CLI)에서 뽑으면 도그푸딩과 겸할 수 있다.

### 3.3 시행과 비결정성 통제

- 에이전트는 비결정적이므로 **(과제 × 군)당 K회(예: K=5) 반복**하고, 단일 값이
  아니라 **분포**로 보고한다(중앙값 + IQR).
- 모델/버전/온도/환경을 **고정·기록**한다. 군 간 유일한 차이는 워크플로우.
- **순서효과** 통제: 군 실행 순서를 과제마다 무작위/카운터밸런싱.
- 동일 git 시작 상태(clean worktree)에서 각 시행을 격리 실행.

### 3.4 채점 블라인딩

- 구현 품질의 주관 점수(루브릭)는 **어느 군 산출물인지 모르는 평가자**(사람 또는
  LLM judge)가 매긴다.
- 자동 지표(수용 테스트 통과율, 시간, 토큰)는 산출물에서 기계적으로 수집.

## 4. QUANTS × GSM 지표

각 차원에 대해 Goal → Signal → Metric. "수집" 열은 어느 페이즈/방법으로 얻는지.

### Q — Quality of the code (스펙 + **구현** 품질) — 헤드라인

목표: vspec 스펙이 *정확하고 완전하며 유지보수 가능한 구현*으로 이어진다.

| Signal               | Metric                                                | 수집             |
| -------------------- | ----------------------------------------------------- | ---------------- |
| 구현 정확성          | **숨겨진 수용 테스트 통과율 %** (1차 결과지표)        | 자동, Phase B    |
| 요구 커버리지        | 문서화된 엣지케이스(Extensions) 중 구현이 처리한 비율 | 자동+루브릭      |
| 누락 엣지케이스 포착 | 과제 서술에 _없던_ 엣지케이스를 구현이 잡은 수        | 루브릭, 블라인드 |
| 재작업률             | green까지 필요한 수정 라운드 수                       | 자동, Phase B    |
| 스펙 내적 품질       | `doctor` error/warn 수, round-trip 통과 (T군 한정)    | 자동, Phase A    |

### A — Attention from engineers (사람 개입 / 흐름) — **사람 부담 패널 ①**

목표: vspec의 가드레일(`doctor`, `suggested_next_actions`)이 에이전트의 헛돎을
줄여, **작업자가 흐름에 끌려 들어가는 일(개입)을 최소화**한다.

| Signal                   | Metric                                                                                 | 수집 |
| ------------------------ | -------------------------------------------------------------------------------------- | ---- |
| **사람 개입** (헤드라인) | **흐름당 사람의 교정/추가 지시 턴 수** (1차 사람부담 지표)                             | 로그 |
| 에이전트 헛돎            | 도구 호출 실패·되돌리기·막다른 길 횟수 _(개입이 준 이유를 설명하는 메커니즘 보조지표)_ | 로그 |
| 가드레일 효과            | 잘못된 시도가 envelope 에러코드로 즉시 차단된 횟수 (T군) _(메커니즘)_                  | 자동 |
| 끊김 없는 진행           | "내버려둬도 굴러갔다" 자기보고 점수(설문)                                              | 설문 |

### N — iNtellectual complexity (인지부하 / 외부화)

목표: Cockburn 지식을 스캐폴딩·검증·`ai-guide`가 떠안아, **사람도 에이전트도**
방법론을 머릿속에 들고 있지 않아도 — 그리고 사람이 스펙을 깊이 파고들지 않아도 —
유효한 스펙이 나온다.

| Signal                   | Metric                                                                                                                          | 수집                |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------- | ------------------- |
| 방법론 외부화            | **fresh-agent 성공률**: `ai-guide`만 읽은 에이전트가 유효 스펙 생산 (사람·에이전트가 방법론을 직접 떠안아야 하는 정도의 대리치) | 자동, Phase A       |
| 구조 오류                | 누락 섹션·잘못된 구조 등 방법론 실수 횟수(=외부화 안 된 부하의 대리지표)                                                        | 자동(doctor)/루브릭 |
| 첫 유효 스펙까지 결정 수 | 사람이 내려야 했던 명령/결정 단계 수                                                                                            | 로그                |
| 추론 부하 대리치         | Phase A 추론 토큰량                                                                                                             | 자동                |

### T — Tempo and velocity (속도·토큰) — **가드레일** (1차 아님)

목표: 어차피 손은 에이전트가 움직이므로 **속도는 후순위**다. 다만 "정확한데 사람이
한참 기다리거나 토큰이 폭증"하는 degenerate case를 잡기 위해 측정은 하되,
헤드라인이 아니라 **회귀-금지 가드레일**로만 쓴다(§5–6).

| Signal           | Metric                                                              | 수집 |
| ---------------- | ------------------------------------------------------------------- | ---- |
| 작성 비용        | Phase A 소요시간·토큰                                               | 자동 |
| 구현 비용        | Phase B 소요시간·토큰                                               | 자동 |
| 전체 시간 (가드) | 첫 통과 구현까지 end-to-end 시간 (A+B 합)                           | 자동 |
| 토큰 비용 (가드) | 전체 흐름 총 토큰($)                                                | 자동 |
| 수렴성           | green까지 반복 횟수 _(사람을 끌어들일 때만 부담으로 침 — A/S 참조)_ | 자동 |

### S — Satisfaction and well-being (검증부담 / 신뢰 / 만족) — **사람 부담 패널 ②③**

목표: 사람이 산출물을 **믿고 바로 수용**할 수 있고(검증부담↓·신뢰↑), vspec 흐름을
선호하며 다시 쓰려 한다.

| Signal                 | Metric                                                                   | 수집      |
| ---------------------- | ------------------------------------------------------------------------ | --------- |
| **검증 부담** (패널 ②) | 수용 전 사람이 읽어야 했던 분량(연 파일/라인·확인 턴) + 검증 노력 Likert | 로그+설문 |
| **신뢰·확신** (패널 ③) | 과제 후 Likert(산출물 신뢰·구현 확신·"믿고 머지 가능했나")               | 설문      |
| 유용성/마찰            | 과제 후 Likert(마찰·"또 쓰겠다")                                         | 설문      |
| 선호                   | 4개 군 강제 순위(forced-choice ranking)                                  | 설문      |
| 정성 테마              | 인터뷰 코딩(좋았던/막혔던 지점)                                          | 인터뷰    |

## 5. 헤드라인 지표와 트레이드오프 규칙

한눈에 보는 1차 지표:

- **품질(Q):** 숨겨진 수용 테스트 통과율 % (목표를 정확히 달성했는가)
- **사람 부담(3패널):** ① 사람 개입 턴 수 · ② 검증 부담 · ③ 신뢰·확신
  — 단일 숫자로 합치지 않고 셋을 **함께** 본다.
- **가드레일(T):** end-to-end 시간·토큰이 파국적으로 회귀하지 않을 것.

**판정은 (Q, 사람 부담) 평면 위에서.** 4개 군을 점으로 찍고 vspec(T군)의 위치를
본다. "통과율은 높은데 사람을 더 귀찮게 함" 또는 "사람 손은 덜 가는데 통과율이
낮음"은 **승리가 아니다.** 속도·토큰은 평면의 축이 아니라 통과/탈락 가드로만 본다.

## 6. 의사결정 기준 (사전 등록)

데이터를 보기 전에 다음을 확정한다. (숫자는 초안 — 합의로 확정)

**vspec 채택(밀기) 조건 — 아래를 모두 만족:**

1. **Q (정확성):** T군 수용 테스트 통과율이 최선의 비-vspec 군 대비 **+X%p 이상**
   (예: +10%p), 통계적으로 분포가 분리.
2. **사람 부담 (1차 비용):** T군 ① 사람 개입 턴 수가 최선의 비-vspec 군 대비
   **유의하게 낮거나 동등**하고, ② 검증 부담·③ 신뢰가 **회귀하지 않음**. (셋을
   합치지 않고 각각 본다 — 하나라도 회귀하면 조건 불충족.)
3. **T 가드레일:** T군 end-to-end 시간·토큰이 최선의 비-vspec 군의 **Z배 이내**
   (예: 1.5배) — 파국적 회귀만 거른다. (속도는 더 이상 1차 조건이 아님.)

**재설계(고치기) 조건:** Q와 사람 부담은 이기는데 T 가드레일을 넘으면 → 작성
페이즈 비용을 줄이는 방향으로 vspec 개선 후 재측정.

**접기(보류) 조건:** Q 우위가 없거나, 사람 부담(개입·검증·신뢰)·N(인지부하)에서
손작성/Superpowers에 밀리면 → 가설 기각 또는 포지셔닝 재고.

## 7. 타당성 위협과 한계

- **에이전트 비결정성** → K회 반복·분포 보고로 완화(§3.3).
- **과제 표본 편향** → vspec에 유리/불리한 과제를 의도적으로 섞고, 과제별로도
  쪼개 본다.
- **평가자 편향** → 블라인드 채점(§3.4).
- **스펙의 재사용 가치 미반영.** 한 번 쓴 스펙은 유지보수·재구현·온보딩에서 또
  값을 한다. 본 설계는 단발 흐름만 보므로 vspec에 **불리하게** 잡힌 셈 → 후속으로
  "스펙 → 2차 변경/유지보수" 시나리오를 별도 측정 가능.
- **Superpowers/손작성 산출물 품질 분산** → 동일 에이전트·동일 모델로 통제.
